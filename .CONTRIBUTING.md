# Contributing to Spotify Backup

Thanks for wanting to contribute! This guide explains how to set up the project, understand the codebase, and submit changes.

## Quick Start

1. **Fork & clone**

   ```bash
   git clone https://github.com/YOUR_USERNAME/spotify-backup.git
   cd spotify-backup
   ```

2. **Install & setup**

   ```bash
   npm install
   cp .env.example .env.local
   # Edit .env.local with your Spotify Client ID
   npm run dev
   ```

3. **Make your changes**
   - Create a feature branch: `git checkout -b feature/my-feature`
   - Make changes (see architecture below)
   - Test locally

4. **Submit a PR**
   - Push to your fork
   - Create a Pull Request with description of changes

## Project Architecture

### Milestones

The app was built in phases:

| Milestone | Feature           | Key Files                                              |
| --------- | ----------------- | ------------------------------------------------------ |
| M1-M7     | Setup + Auth      | `AuthContext.tsx`, `spotifyAuth.ts`, `pkce.ts`         |
| M8        | Backup & Import   | `backupFormat.ts`, `validateBackup.ts`, `useBackup.ts` |
| M9        | Playlist fetch    | `usePlaylists.ts`, `spotifyApi.ts`                     |
| M10       | Create playlists  | `usePlaylistCreator.ts`, `restoreApi.ts`               |
| M11       | Add tracks        | `useTrackRestorer.ts`                                  |
| M12       | Save liked songs  | `useLikedSongsRestorer.ts`                             |
| M13       | Results dashboard | `ResultsPanel.tsx`, `resultsExport.ts`                 |
| M14       | Cleanup & docs    | `errorCodes.ts`, `README.md`                           |

### Data Flow

```
AuthContext                          BackupContext
├── source account (tokens + user)   ├── imported backup JSON
└── destination account              ├── playlistMap (source → dest IDs)
                                     └── restore results

HomePage
├── useAuth() → get accounts
├── usePlaylists() → fetch source playlists
├── useLikedSongs() → fetch liked songs
└── useBackup() → orchestrate backup

Restore Flow:
1. ImportPanel → loads backup JSON into BackupContext
2. RestorePanel → coordinates three steps:
   - usePlaylistCreator → create playlists in destination
   - useTrackRestorer → add tracks to playlists
   - useLikedSongsRestorer → save liked songs
3. ResultsPanel → show final stats
```

### Adding New Features

#### 1. API Call Needed?

- Read operation: Add to `spotifyApi.ts`
- Write operation: Add to `restoreApi.ts`

Example:

```typescript
// lib/spotifyApi.ts
export async function myNewFunction(
  token: string,
  param: string,
): Promise<MyType> {
  const res = await fetch(`${BASE}/endpoint`, {
    headers: authHeader(token),
  });
  if (!res.ok) throw new Error("my_error_code: details");
  return res.json() as Promise<MyType>;
}
```

#### 2. Need State Management?

- User data: `AuthContext.tsx`
- Backup data: `BackupContext.tsx`

Example:

```typescript
// context/MyContext.tsx
export const MyContext = createContext<MyContextValue | null>(null)

export function MyProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState(/* ... */)
  return <MyContext.Provider value={{ state }}>{children}</MyContext.Provider>
}

// hooks/useMyContext.ts
export function useMyContext(): MyContextValue {
  const context = useContext(MyContext)
  if (!context) throw new Error('useMyContext must be used within MyProvider')
  return context
}
```

#### 3. Need a Hook?

Place in `hooks/` folder and name it `useMyFeature.ts`.

Example:

```typescript
// hooks/useMyFeature.ts
import { useState, useCallback } from "react";
import { myApiFunction } from "../lib/spotifyApi";
import { useAuth } from "./useAuth";

export interface UseMyFeatureResult {
  status: "idle" | "loading" | "done" | "error";
  result: MyResult | null;
  error: string | null;
  start: () => Promise<void>;
}

export function useMyFeature(): UseMyFeatureResult {
  const { source } = useAuth();
  const [status, setStatus] = useState<"idle" | "loading" | "done" | "error">(
    "idle",
  );
  const [result, setResult] = useState<MyResult | null>(null);
  const [error, setError] = useState<string | null>(null);

  const start = useCallback(async () => {
    try {
      setStatus("loading");
      const data = await myApiFunction();
      setResult(data);
      setStatus("done");
    } catch (err) {
      setError(err instanceof Error ? err.message : String(err));
      setStatus("error");
    }
  }, []);

  return { status, result, error, start };
}
```

#### 4. Need a Component?

Create folder in `components/` with:

- `MyComponent.tsx` — the React component
- `MyComponent.css` — styles
- Use atomic naming: `component__element--modifier`

Example:

```typescript
// components/MyComponent/MyComponent.tsx
import './MyComponent.css'

interface MyComponentProps {
  title: string
  onClick: () => void
}

export default function MyComponent({ title, onClick }: MyComponentProps) {
  return (
    <div className="my-component">
      <h2 className="my-component__title">{title}</h2>
      <button className="my-component__btn" onClick={onClick}>
        Click me
      </button>
    </div>
  )
}
```


## Error Handling

**Always use error codes!** See `lib/errorCodes.ts`:

```typescript
// Instead of:
throw new Error("something went wrong");

// Do this:
throw new Error("error_code: human readable message");
```

Examples:

- `auth_redirect_uri_mismatch: redirect_uri does not match...`
- `api_rate_limited: Spotify rate limit hit...`
- `file_invalid_json: Backup file is not valid JSON...`

The `parseErrorMessage()` function automatically converts these to user-friendly messages with recovery hints.

## Code Style

### TypeScript

- Always use strict types (no `any`)
- Use `interface` for public APIs, `type` for local types
- Document complex functions with JSDoc

### React

- Use functional components + hooks
- Prefer `useCallback` for memoized functions
- Keep components small (< 200 lines)

### CSS

- Use CSS variables (`var(--color-accent)`)
- Use BEM naming (`component__element--modifier`)
- No external UI libraries
- Mobile-first, responsive design

### File Organization

```
good/
├── components/
│   └── MyComponent/
│       ├── MyComponent.tsx    ← Component only
│       └── MyComponent.css    ← Styles
├── hooks/
│   └── useMyHook.ts          ← One hook per file
├── lib/
│   ├── myApi.ts              ← API functions
│   └── myUtils.ts            ← Utilities
└── pages/
    └── MyPage.tsx            ← Full page component
```

## Testing

Run the app locally and manually test:

1. **Authentication flow**

   ```bash
   npm run dev
   # Try logging in with two different accounts
   # Check browser console for errors
   ```

2. **Backup flow**
   - Connect source account
   - Click "Backup"
   - Verify JSON downloaded

3. **Restore flow**
   - Import the JSON backup
   - Connect destination account
   - Run restore steps
   - Check results panel

4. **Error cases**
   - Clear localStorage and try logging in (test PKCE recovery)
   - Change redirect URI and try logging in (test redirect_uri mismatch error)
   - Clear browser storage during backup (test error recovery)

## Performance Tips

- **Images**: Optimize before committing (compress PNGs/JPGs)
- **Bundles**: Check `npm run build` output size
- **API calls**: Batch when possible, use 300ms delays between writes
- **UI**: Use `useCallback` to avoid unnecessary re-renders
- **Local storage**: Only store essential data (tokens, user profile)

---

## Security Guidelines

- ✅ Never hard-code secrets (use `.env`)
- ✅ Always validate user input (file uploads, etc.)
- ✅ Use PKCE for OAuth (never send client secret)
- ✅ Clear sensitive data on logout
- ✅ Use HTTPS in production
- ❌ Don't log tokens to console
- ❌ Don't send sensitive data to analytics

## Submitting a PR

1. **Title**: `feat: description` or `fix: description`
2. **Description**: What changed and why
3. **Testing**: How you tested it
4. **Checklist**:
   - [ ] Code follows style guide
   - [ ] Added comments for complex logic
   - [ ] Tested locally
   - [ ] No console errors

Example:

```
feat: add export to CSV format

Adds ability to export backup as CSV instead of just JSON.
Users can now open backup in Excel/Sheets.

Tested by:
- Exporting a backup with 500+ tracks
- Verified CSV format is valid
- Tested with Excel, Sheets, and Google Sheets

No breaking changes, fully backwards compatible.
```

## Release Process

Releases are handled by the maintainer:

1. Update version in `package.json`
2. Create git tag: `git tag v1.2.3`
3. Push tag: `git push origin v1.2.3`
4. Vercel auto-deploys when pushed to `main`
